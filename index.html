接口与抽象类的区别

接口与抽象类的都是对底层的一种抽象，都是实现多态的一种方式（动态绑定），且都不能被实例化。

1、属性区别
  • 接口是接口，只能被子接口继承、被子类实现
  • 抽象类是类，只能被子类继承
2、访问标志区别

  • 接口的访问标志为：interface、abstract

  • 抽象类访问标志为：abstract、super

3、字段区别
  • 接口中只能含有公有的常量字段 public static final
  • 抽象类可以含有任意类型字段，如同普通的类一致
4、方法区别
  • 接口中只能含有3种类型的方法【公有的抽象方法（public abstract）、公有的静态方法（public static）、公有的默认方法（public）】//实际上在jvm中默认方法也就是删除了abstract访问标志而已
  • 抽象类中的方法与普通类一致，不过可以添加abstract类型的类。

hashcode 与 equals

重写equals时为什么一定要重写hashcode

  • 首先我们来看看两个函数的作用：hashcode（）其实是一个散列函数，把不同对象通过散列函数，转换为一个code值，用于hashset 或者 hashmap，这就需要满足相同的对象肯定在同一个桶里面，如果equals改变而hashcode不改变，就会有可能造成相同的对象在不同的桶里面。

  • 值传递：传递的是本身的一个副本，传递过程有个拷贝的
  • 引用传递：传递的是本身的一个引用

自动拆箱与装箱

为了更好融入java对象系统，java5就增加了自动的拆箱与装箱功能。值得一提的是，该功能由编译器完成的，jvm完全没有变化。

在Integer类文件中，有两个方法【静态方法：value（）】把int型转换为Integer, 非静态intVal（）获取Integer的值。 为了减少内存消耗，这里添加了一个缓存区优化装箱功能。当装箱的为（-128 —- +127）时，返回缓存后的值。

String类型

String类底层由char型数组所构成，含有两个字段：一个是字符串数组，一个是hash码

可变性
  • String不可变：上图可知，字符串数组为final类型，是不可改变的
  • StringBuilder、StringBuffer可变：继承与AbstractStringBuilder，底层为非final字符数组，因此是可变的

线程安全性

  • String线程安全：String中字符数组是不可变的，只可读因此是线程安全的
  • StringBuffered线程安全：字符数组是可变的，但在方法中添加由同步锁，是线程安全的
  • StringBuilder线程不安全:字符数组是可变的，在方法中也没有添加同步锁，是线程不安全的

性能

  • 数据量少用String，数据量大且单线程用StringBuilder，数据量大且多线程使用StringBuffer

String hash字段

String中的hash字段能够加速HashMap的查询速度

同步锁

synchronized是JVM中的一个访问控制标签，使用它的方式大概有两种，一个是同步方法，一个同步代码块，这两种方式，虽然使用方式不同，但是实现的原理都是基于管程的进入与退出实现的。

  • 同步代码块：java编译器会将代码块头尾添加monitorenter 与 monitorexit，分别为管程进入与退出，管程与对象绑定，每个对象都有管程，管程中有计数器，当计数器不为0，且该管程不属于该线程是不允许进入的

  • 同步方法：java编译器不会显示使用monitorenter与monitorexit，但是在方法字段中会有synchronized，在方法调用和返回时会执行管程进入与退出。

5、面向连接的含义

​ 面向连接的服务就是通信双方在通信时,要事先建立一条通信线路,其过程有建立连接、使用连接和释放连接三个过程。TCP协议就是一种面向连接服务的协议,电话系统是一个面向连接的模式。

6、为什么需要电路交换

​ 由于经济上的原因，中继线路总是大大少于用户线路，并且为所有用户所共享。如A城的用户到B城用户，通过A城交换机与B城交换机进行相连，大大减少了N^2级别的线路。那么，当我们占用了一条中继线路以后，即使我们不传送信息，别人也不能使用，这就是电路交换最主要的缺点。

7、电路交换与分组交换的优缺点

​ 从电路交换的工作原理看出，电路交换会占用固定带宽，即上文6点所提到的（当我们占用了一条中继线路以后，即使我们不传送信息，别人也不能使用），因而限制了在线路上的流量以及连接数量 。

​ 电路交换常与分组交换进行比较。其主要不同之处在于：分组交换的通信线路并不专用于源与目的地间的信息传输。在要求数据按先后顺序且以恒定速率快速传输的情况下，使用电路交换是较为理想的选择。

​ 因此，当传输实时数据时，诸如音频和视频；或当服务质量（QOS）要求较高时，通常使用电路交换网络。分组交换在数据传输方面具有更强的的效能，可以预防传输过程（如 e-mail 信息和 Web 页面）中的延迟和抖动现象。

​ 此外，分组交换由于存在路由器分组转发的情况，因此就存在分组转发延时的问题。

8、分组交换与报文交换的优缺点

​ 与7点相比，电路交换与分组交换的对比层面是在能否动态分配带宽，提高传输效率，而报文交换（电报的传输）与分组交换主要的比较是在分组与不分组的层面，这方面分组交换，为每一个分组独立地选择合适的转发路由，使得数据传输更加灵活。

16、5层网络体系结构，各层的功能

  • 应用层：直接为用户的应用进程提供服务，如http协议，为的是web进程提供服务

  • 运输层：向上一层的进行通信的两个进程之间提供一个可靠的端到端服务。

  • 网络层：选择合适的路由，使发送站的运输层所传下来的分组能够正确无误地按照地址找到目的站，并交付给目的站的运输层。

  • 数据链路层：在两个相邻结点间的线路上无差错地传送以帧（frame）为单位的数据。每一帧包括数据和必要的控制信息。

  • 物理层：传送比特流

17、带宽的理解**

​ 以前对于带宽是不理解的，因为电子方面也有带宽，计算机网络里也有带宽，以为这两种带宽是完全不相干的东西。

​ 电子通信方面的带宽单位为HZ，其实也就是通频带的范围多大。下面的图如果学过电子的肯定非常熟悉，信号放大器的放大倍数随着频率的高低而变化，当降低到中心频滤放大倍数的0.707倍时中间的频率为带宽。这个带宽用于传输数字信号时，也就是1s内最多能够传输带宽个0或者1，这个带宽还能够转换为bit/s，也就是链路的传输的速率，虽然单位不同，但是本质是一样的。

35、造成路由环路的原因

  • 配置错误，即静态路由配置错误，造成一个环，这时只能靠ip数据报中的TTL来实现丢弃该数据报；
  • 聚合了不存在的路由，这其实也是路由配置问题，当聚合不存在的路由时，配合默认路由可以造成一个路由环路；使用黑洞路由可以解决该问题
  • 网络故障，当出现网络故障时，路由表的记录会给删除，配合默认路由，会造成路由环路问题。实际中，当路由有问题时，路由表中的对应黑洞路由会生效，可以避免该问题(需要事先配置好黑洞路由，之后才会这样生效);
  • 值得注意的是，如果路由器开启RIP协议，并且RIP协议没有开启水平分割或者毒化逆转，当路由端口或者链路出现问题时，也将有会造成路由环路，虽然实际中并没有不开启水平分割或者毒化逆转的情况。

47、为什么需要三次握手而两次握手可行吗？

正常的TCP连接

这个问题可以从两个角度来回答

如下图所示，如果使用两次握手则可以建立连接的话，当有一个SYN包网络滞后到达服务器，不需要经过客户端确认，则也会形成连接，就造成了资源的浪费。

48、四报文挥手

需要四次的原因和建立需要三次的原因差不多，因为释放也需要确认，但是双方是全双工的，A想要释放（停止发送），B不一定想要释放（还有数据要发送给A）。值得说明的是，四次挥手不一定是指要四个报文，有种例子，当A->B释放连接请求时，B之后也没有数据要给A，那么ACK+FIN的报文可以一起发送出去，这样就变成了三报文挥手了。

50、TCP可靠实现机制总结

  • 校验和：

TCP首部中有校验和字段，用于校验TCP首部与数据载荷部分是否出错。

  • 确认应答机制（ACK）

TCP通过也确认应答机制实现可靠传输，在TCP首部有一个标志位ACK，此标准表示首部中的确认号字段是否有效，如果发送方收到了已发送数据的确认报文，则会继续发送数据，如果等待了一段时间还没有收到确认报文就会启动重传机制。

  • 超时重传

发送数据超过时间RTO时，则会触发超时重发机制，重新发送数据。

超时有可能是发送的包丢了（或者在网络中滞留时间太长）

超时有可能是确认的包丢了（或者在网络中滞留时间太长）

RTO的计算是比较复杂的，动态的根据网络情况进行更改

  • 连接管理

即三报文握手和四报文挥手机制

  • 流量控制

TCP首部中有窗口字段，代表着发送本报文方接收窗口的大小，而发送数据方需要根据这个接收窗口，修改发送窗口大小，即流量控制。

  • 拥塞控制

流量控制是解决两台主机之间因传输速率不匹配导致的丢包问题，而拥塞主要是解决因网络环境拥堵所造成丢包的问题，如网络比较拥堵，造成报文在网络中长时间滞留，导致触发超时重发机制，这就大大降低网络的吞吐量。

拥塞控制2种机制：慢开始与拥塞避免

因为丢包造成的超时重传可能并不是由于网络拥塞导致的，一超时就启动慢开始机制，可能会降低效率

所以添加了另外两种机制：快重传与快恢复

  • 慢开始：初始cwnd设置为1，一旦接收到确认报文，cwnd窗口扩大两倍，cwnd必须小于慢开始门限值ssthresh。
  • 拥塞避免：大于ssthresh，cwnd每收到一个确认报文则+1
  • 快重传：当收到3次重复确认，则触发快重传机制
  • 快恢复：按照以往，收到一次超时重传，慢开始门限更新为触发超时重传的一半，cwnd设置为1，重新启动慢开始机制，而如果是触发了快重传，则不启动慢开始，启动快恢复，慢开始门限更新为触发超时重传的一半，cwnd设置为当前门限值，则触发拥塞避免算法。

  • git commit -m “flag”:flag为标签

git remote:

主要用于配置远端仓库的命令

  • git remote -v：查看远端地址，没有输出则为空

  • git remote add [name] [url] :添加远端地址name为url

  • git remote rm [name]:删除远端地址name

git add

  • git add . :将修改的所有文件推入缓存区

git push

用于将暂存区的文件上传至云端

  • git push [name] [branch]:name为远端仓库，branch为远端分支与本地分支名称

git clone

下载远端仓库的内容，可以是整个仓库也可以是一个分支

  • git clone [url]:将url所代表的仓库全部clone下来
  • git clone -b [branch] [url]:branch 为分支名称，url为远端仓库
